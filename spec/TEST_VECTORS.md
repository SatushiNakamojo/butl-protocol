# BUTL Test Vectors

## Deterministic Test Cases for Cross-Implementation Verification

These test vectors use fixed seeds and inputs. Any correct BUTL implementation in any language will produce identical outputs for every vector. Use these to verify that an implementation is compatible with the reference and interoperable with other implementations.

All values in this document were generated by running the BUTL MVP (`butl_mvp_v12.py`) with the exact seeds specified. They are reproducible on any machine.

---

## How to Use This Document

1. Implement BUTL in your language.
2. Feed the exact input values below into your implementation.
3. Compare your outputs to the expected values.
4. If every output matches, your implementation is compatible. If any differ, there is a bug.

**Important conventions:**
- All hex values are lowercase.
- All hashing uses SHA-256 as defined in FIPS 180-4.
- All keys use the secp256k1 curve.
- All strings are UTF-8 encoded before hashing.
- "seed" always means the 32-byte output of `SHA-256(ascii_string)`.

## Test Vector 1: Key Generation (Sender)

This vector verifies that key generation (seed -> private key -> public key -> address) is implemented correctly.

### Input

```
seed_string = "butl-test-vector-sender-0"
seed = SHA-256(UTF-8("butl-test-vector-sender-0"))
```

### Expected Output

```
seed_hex:             61a6eae659ceff3e6bd3570a98ca319660aa451e65f15325197c7b3a37b59552
private_key (int):    44169264144220881511725783510596205141403330907180823116918147268650231764307
public_key_compressed: 03aa6d590b9689180908401aaeff9e116526cbe90d0f414e851d09b4e4b8abacfc
bitcoin_address:      14Zzmx3Mi773r4U9FjKHbtNJw1TVTZjyho
```

### Derivation Steps

```
1. seed = SHA-256("butl-test-vector-sender-0")
       = 61a6eae659ceff3e6bd3570a98ca319660aa451e65f15325197c7b3a37b59552

2. private_key = int(seed) mod (n - 1) + 1
   where n = FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141

3. public_key_point = private_key x G
   (secp256k1 scalar multiplication)

4. compressed = 03 || public_key_point.x
   (03 because y-coordinate is odd)
   = 03aa6d590b9689180908401aaeff9e116526cbe90d0f414e851d09b4e4b8abacfc

5. address = Base58Check(0x00 || RIPEMD-160(SHA-256(compressed)))
           = 14Zzmx3Mi773r4U9FjKHbtNJw1TVTZjyho
```

### What to Verify

- Your `seed_hex` matches after hashing the string.
- Your `public_key_compressed` matches (this confirms scalar multiplication is correct).
- Your `bitcoin_address` matches (this confirms SHA-256 -> RIPEMD-160 -> Base58Check pipeline).

## Test Vector 5: ECDH Shared Secret

Verifies that ECDH key agreement produces identical shared secrets from both sides.

### Input

```
sender_private_key:   (from Test Vector 1)
receiver_public_key:  024f2acadea7c3ee274f0f5ec874beb01deeba993a36de5ff2453427a74cc16edb

receiver_private_key: (from Test Vector 3)
sender_public_key:    03aa6d590b9689180908401aaeff9e116526cbe90d0f414e851d09b4e4b8abacfc
```

### Computation

```
sender_computes:   shared = SHA-256( (sender_priv x receiver_pub).x )
receiver_computes: shared = SHA-256( (receiver_priv x sender_pub).x )
```

### Expected Output

```
shared_secret_sender:   b0ef30fcdb2e43e10df3d89f0ae75761a8e7ed9e660e7b8afbfef79ed9accac3
shared_secret_receiver: b0ef30fcdb2e43e10df3d89f0ae75761a8e7ed9e660e7b8afbfef79ed9accac3
secrets_match:          true
```

### What to Verify

- Both computations produce the exact same 32-byte hex string.
- This confirms ECDH commutativity: `sender_priv x receiver_pub == receiver_priv x sender_pub` after hashing.

### Wrong Receiver (Negative Test)

```
wrong_seed = SHA-256(UTF-8("butl-test-vector-wrong-person"))
wrong_private_key = keygen(wrong_seed).private_key
wrong_shared = SHA-256( (wrong_priv x sender_pub).x )

wrong_shared != shared_secret_sender    (MUST be different)
```

A wrong receiver computes a different shared secret. AES-GCM decryption will fail with AUTH FAILED.

## Test Vector 6: ECDSA Signature and Verification

Verifies that ECDSA signing and verification work correctly.

### Input

```
private_key:  (from Test Vector 1)
public_key:   03aa6d590b9689180908401aaeff9e116526cbe90d0f414e851d09b4e4b8abacfc
message:      UTF-8("BUTL-test-vector-signature-message")
message_hash: SHA-256(message)
```

### Process

```
1. signature = ECDSA_SIGN(private_key, message_hash)
2. valid = ECDSA_VERIFY(public_key, signature, message_hash)        -> true
3. wrong = ECDSA_VERIFY(receiver_public_key, signature, message_hash) -> false
```

### Expected Output

```
correct_key_verify: true
wrong_key_verify:   false
```

### Note on Signature Determinism

Standard ECDSA includes a random nonce `k`, so signature bytes differ each run. To produce byte-level deterministic signatures, use **RFC 6979** deterministic ECDSA (deriving `k` from the private key and message hash). Most production libraries support this.

If your library uses random `k`, you can still validate: sign with your library, then verify with your library. The verification result (true/false) is deterministic even if the signature bytes are not.

## Test Vector 7: Address Chaining

Verifies that chain proof creation and verification work correctly.

### Input

```
previous_keypair:  (from Test Vector 1)
   private_key_0:  (TV1 private key)
   public_key_0:   03aa6d590b9689180908401aaeff9e116526cbe90d0f414e851d09b4e4b8abacfc
   address_0:      14Zzmx3Mi773r4U9FjKHbtNJw1TVTZjyho

current_keypair:   (from Test Vector 2)
   address_1:      1DdB2aj2od12RTHpm5CPCVVuH49veWff7v
```

### Computation

```
chain_hash  = SHA-256(UTF-8("1DdB2aj2od12RTHpm5CPCVVuH49veWff7v"))
chain_proof = ECDSA_SIGN(private_key_0, chain_hash)
```

### Expected Output

```
chain_hash:                  c0eaa0157d52330cd03f37fda31ff5c4d38bac5fe8d8abbad2a164ff37ded0f2
chain_proof_verify_correct:  true   (verified with public_key_0)
chain_proof_verify_wrong:    false  (verified with any other public key)
```

### Thread ID

```
thread_id = SHA-256_HEX(UTF-8("14Zzmx3Mi773r4U9FjKHbtNJw1TVTZjyho"))
          = 8aa0c2306d46c74e512b30b6967f9e7e5442e38ab78126fd935f0d10a221249b
```

### What to Verify

- `chain_hash` matches exactly.
- Chain proof verifies with the correct (previous) public key.
- Chain proof does NOT verify with any other public key.
- `thread_id` matches exactly (constant for all messages in this chain).

Verifies that the canonical signing payload is constructed correctly and produces the expected hash.

### Input

```
version:          1
sender:           14Zzmx3Mi773r4U9FjKHbtNJw1TVTZjyho

receiver_pubkey:  024f2acadea7c3ee274f0f5ec874beb01deeba993a36de5ff2453427a74cc16edb
block_height:     890412
block_hash:       00000000000000000002a7c4c1e48d76f0593d3eabc1c3d1f92c4e5a6b8c7d9e
payload_hash:     20016ed6d12f5411a5dd5749cfb2db43412402db661d6ba1d56f62f660bfb3b1
prev_addr:        (empty string)
nonce:            (empty string)
```

### Constructed Payload

```
BUTL-Version:1
BUTL-Sender:14Zzmx3Mi773r4U9FjKHbtNJw1TVTZjyho
BUTL-SenderPubKey:03aa6d590b9689180908401aaeff9e116526cbe90d0f414e851d09b4e4b8abacfc

BUTL-BlockHash:00000000000000000002a7c4c1e48d76f0593d3eabc1c3d1f92c4e5a6b8c7d9e

BUTL-Nonce:
```

Note: Fields separated by `\n` (0x0A). No spaces around colons. Empty strings for PrevAddr and Nonce (fields present, values empty).

### Expected Output

```

```

### What to Verify

- Your constructed canonical payload matches the exact string above (including empty PrevAddr and Nonce lines).
- The SHA-256 hash of the UTF-8 encoding of this string matches `b960...1864`.
- This hash is what gets signed with ECDSA for `BUTL-Signature`.

## Test Vector 11: Consistency Check

Verifies that the pubkey-to-address consistency check catches mismatches.

### Input (Matched - Should Pass)

```
sender_pubkey: 03aa6d590b9689180908401aaeff9e116526cbe90d0f414e851d09b4e4b8abacfc
sender_addr:   14Zzmx3Mi773r4U9FjKHbtNJw1TVTZjyho

derived = Base58Check(0x00 || RIPEMD-160(SHA-256(sender_pubkey)))
derived == sender_addr -> true -> PASS
```

### Input (Mismatched - Should Fail)

```
sender_pubkey: 03aa6d590b9689180908401aaeff9e116526cbe90d0f414e851d09b4e4b8abacfc
sender_addr:   16J5UTAiCJYFfWYN35zPZEHP6zdfZS71iz  <- this is the RECEIVER's address, not the sender's

derived = Base58Check(0x00 || RIPEMD-160(SHA-256(sender_pubkey)))
derived != sender_addr -> false -> REJECT
```

### What to Verify

- Matched pair passes the consistency check.
- Mismatched pair is rejected at Step 1 (Structural Validation).
